# Полная шпаргалка по работе с ветками в Git и Pull Requests
---
## 1. Работа с ветками

### Создание новой ветки и переключение на неё
git checkout -b имя_ветки

### Просто создать ветку, без переключения
git branch имя_ветки

### Переключение между ветками
git checkout имя_ветки  
или (Git 2.23+)  
git switch имя_ветки

### Удаление веток
git branch -d имя_ветки # мягкое удаление (если ветка слита)  
git branch -D имя_ветки # принудительное удаление
git push origin --delete имя_ветки # удаление удалённую ветку

### Просмотр веток
git branch # Локальные
git branch -a # Список всех веток (локальные + удалённые)

## 2. Объединение веток (слияние)
### Слияние ветки в текущую
git checkout main  
git merge имя_ветки

### Использование rebase (перемещение коммитов)
git checkout имя_ветки  
git rebase main
- Помогает сделать историю чище, перемещая коммиты ветки поверх основного дерева.

## 3. Именование веток: лучшие практики

- Формат имен: `<тип>/<тема>`
- Примеры типов:
  - `feat` — новая функциональность (feature)
  - `fix` — исправление ошибки
  - `chore` — технические задачи, не относящиеся к функционалу (настройки, рефакторинг)
  - `hotfix` — срочное исправление на продакшене
  - `docs` — документация, изменения в документации
- Примеры:
  - feat/login-form
  - fix/button-alignment 
  - chore/update-deps
  - hotfix/login-crash 
  - docs/readme-update

- При использовании трекера задач, добавляйте номер задачи:

feat/DEV-123-add-login-form  
fix/BUG-456-fix-button-alignment

- Имена веток должны быть в нижнем регистре, слова разделяются дефисом.

## 4. Работа с удалёнными ветками

### Удалить локальную ветку
git branch -d имя_ветки 
### Удалить удалённую ветку
git push origin --delete имя_ветки
### Синхронизировать локальный репозиторий с удалённым
git fetch
### Создать локальную ветку, отслеживающую удалённую
git checkout -t origin/имя_ветки
### Получить изменения с удалённого и автоматом слить (включает fetch + merge)
git pull

### Получить изменения с удалённого и применить rebase (чистая история)
git pull --rebase

## 5. Работа с Pull Request (PR)
### Шаги для создания Pull Request:
1. Создать новую ветку для работы
git checkout -b feat/my-feature
2. Сделать коммиты с изменениями
git add .
git commit -m "feat: добавить новую функцию"
3. Запушить ветку на удалённый репозиторий
git push origin feat/my-feature
4. Открыть репозиторий на GitHub (или другом хостинге) и перейти в раздел **Pull Requests**.
5. Нажать **New Pull Request** и выбрать:
- Ветка назначения (обычно `main` или `master`)
- Ветка с вашими изменениями (например, `feat/my-feature`)

6. Создать PR, указав название и описание.

7. Дождаться ревью и слияния (merge) либо попросить изменить.

### Пример ветки для PR:
feat/username-123-new-login
fix/username-456-fix-bug

## 6. Работа с форками

1. На GitHub создайте Fork нужного репозитория.

2. Клонируйте Fork:
git clone https://github.com/yourusername/project.git
cd project
3. Добавьте оригинальный репозиторий как удалённый `upstream`:
git remote add upstream https://github.com/original/project.git
4. Чтобы синхронизировать Fork с оригиналом:

git fetch upstream  
git checkout main  
git merge upstream/main  
git push origin main

5. Работайте в ветках, делайте изменения, коммитьте, пушьте в Fork.

6. Создавайте Pull Request из ветки вашего Fork в репозиторий оригинала.

## 7. Советы по работе с ветками и PR

- Следуйте принятому в команде стилю именования веток.
- Регулярно делайте обновление веток с основной (rebase или merge).
- Удаляйте локальные и удалённые ветки после слияния.
- В PR оставляйте информативные описания и связывайте с задачами.
- Используйте ревью, чтобы поддерживать качество кода.

---

С этой шпаргалкой вы сможете уверенно управлять ветками, организовывать работу над функционалом и взаимодействовать через pull requests и форки.
